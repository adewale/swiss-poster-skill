Agents vs Workflows.

Same harness, two lifecycles. An Agent loops and lives on; a Workflow runs once and returns. Follow each from definition to its end.
Agent
model-driven · open-ended · persistent
export default createAgent(() => ({ model, tools, skills, sandbox }))
01
Define
src/agents/triage.ts — a createAgent() default export
02
Serve, headless
POST /agents/triage/:id · channels · sessions
03
Message opens a session
the persistent "default" session accumulates history
04
Loop ↺
the model: reason → act → observe (tools · sandbox), until done
05
Persist
event-sourced Durable Streams — accepted work is never lost
↻
Lives on
interrupted sessions resume · clients reconnect · no final value — it waits for the next message
Workflow
code-driven · finite · typed result
export async function run({ init, payload }: FlueContext<T>)
01
Define
src/workflows/triage.ts — export run() (file = name)
02
Invoke
POST /workflows/triage or flue run → admission
03
Get a runId
202 { runId, streamUrl, offset } — one durable run
04
Execute to completion
your code: init → session → prompt(input, { result })
05
Return
typed response.data · ?wait=result to block
■
Done — run recorded
inspect later: flue logs <runId> · GET /runs/<runId> · client.runs.get()
vs
Same harness underneath
init(agent) → harness.session() → session.prompt(input, { result? })
Agent — the model drives and the session lives on. Reach for it when the path isn't known up front.
Workflow — your code drives to a typed result and a recorded run. Reach for it when the steps are known.
