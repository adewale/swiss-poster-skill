Agents vs Workflows.

Flue runs work two ways on the same harness. The difference is one question: who drives the loop — the model, or your code?
Agents
model-driven · open-ended · persistent
The model drives — reason → act → observe, looping until the task is done.
// src/agents/triage.ts export default createAgent(() => ({ model: 'anthropic/claude-sonnet-4-6', tools, skills, sandbox: local(), instructions, }));
Driver
the model — the loop
Shape
open-ended · multi-turn
Trigger
messages · sessions · channels
Returns
a persistent, resumable session
Use when
the path isn't known up front — conversation, triage, evolving goals.
Workflows
code-driven · finite · typed result
Your code drives — call prompt() where you choose; return a typed value.
// src/workflows/triage.ts export async function run( { init, payload }: FlueContext<{ ticket: string }>, ) { const s = await (await init(agent)).session(); const r = await s.prompt(payload.ticket, { result: v.object({ priority: v.picklist([…]), summary: v.string() }), }); return r.data; // typed · finite }
Driver
your TypeScript
Shape
finite · run-to-completion
Trigger
POST /workflows/:name → { runId }
Returns
response.data — a typed result
Use when
the steps are known — pipelines, batch jobs, structured extraction.
vs
Same harness underneath
init(agent) → harness.session() → session.prompt(input, { result? })
Same model, tools, skills, and sandbox. An Agent lets the model decide the next step; a Workflow lets your code decide — and hands back a typed result with a runId.
