Inside Eve.

Vercel's “Next.js for agents,” read from the repo and docs. The agent is a directory; Vercel is the runtime. Capabilities are files in conventional slots — and the framework leans on five Vercel services underneath.
The agent is a directory agent/
agent/ ├─ agent.ts defineAgent({ model }) · runtime config ├─ instructions.md the system prompt ├─ tools/ │ └─ get_weather.ts defineTool() → tool “get_weather” ├─ skills/ .md playbooks, loaded on demand ├─ subagents/ an agent one level down ├─ channels/ defineChannel() · http · slack · discord ├─ schedules/ defineSchedule({ cron }) ├─ connections/ defineMcpClientConnection() └─ evals/ defineEval() · t.check()
Path-derived naming: agent/tools/get_weather.ts becomes the get_weather tool. No registry, no wiring — the folder is the agent.

Stands on five Vercel services the runtime
Workflows
Durable sessions — checkpoint, survive a crash or a deploy, resume where it stopped.
Sandbox
Isolated compute. Adapters: Vercel Sandbox in prod · Docker / microsandbox locally.
AI Gateway
Model routing + provider fallbacks. OIDC, no API keys — e.g. openai/gpt-5.4-mini.
Connect
OAuth tokens & API keys for the external services an agent touches.
Observability
Agent Runs in the dashboard — turns, tools, tokens, timing; OTel spans export anywhere.
What's distinctive vs the field
Durable pause
A session can pause and survive a deploy, then resume exactly where it stopped.
HITL approval
Any action can require approval — it waits indefinitely, consuming no compute.
Built-in evals
Scored suites, local or in CI — defineEval with t.check().
Subagents
Just a folder under subagents/; the parent calls it like a tool.

The takeaway: Eve is the framework; Vercel is the runtime. Brilliant DX if you live on Vercel — the inverse of a portable harness.
