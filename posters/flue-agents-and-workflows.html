If you're not the model,
you're the harness.

The model reasons. The harness is everything around it — tools, skills, a sandbox, sessions, the loop. Most teams rebuild that scaffolding for every agent. Flue makes the harness the framework, and lets you choose how much the model drives it: Agents and Workflows are two settings on the same dial.
One harness · two ways to drive it who decides what happens next?
Agents and Workflows are the same harness, driven differently. Slide the model toward your code (a Workflow returns a typed result; an Agent runs a persistent loop and can spawn declared sub-agents). The harness — and Pi, the loop it's built on — never changes.

The same harness, two files init → session → prompt
Workflow — your code calls the model
// src/workflows/triage.ts
export async function run({ init, payload }) {
  const s = await (await init(agent)).session();
  const r = await s.prompt(payload, { result });
  return r.data;            // typed · finite
}
Agent — the model calls your code
// src/agents/triage.ts
export default createAgent(() => ({
  model, tools, skills,
  sandbox: local(),
  instructions,            // the model drives
}));
“Why not just have the model generate the boilerplate?”
Because the boilerplate was never the hard part — the harness is. A sandbox that's a primitive (not glue you rewrite per backend), runs that resume after a crash, typed results you can test, one agent that deploys anywhere. That's the layer you'd rebuild for every agent and get subtly wrong — and the less code you write, the less there is to break.
If you're not the model, you're the harness — so own a good one.
