one definition · two ways to run it
The agent is the unit · the workflow runs it

Define an agent once. Run it two ways.

An agent is the unit: a model plus its harness — tools, skills, a sandbox. A workflow is code that calls agents and sequences them to a result. Serve the agent and the model drives the loop; call it from a workflow and your code drives. The agent and the harness are identical either way.
The workflow imports the agent same agent, two files
src/agents/triage.ts — the unit
export default createAgent(() => ({ model, tools, skills, sandbox: local() }));
// served on its own — the model drives
src/workflows/triage.ts — code that runs it
import triageAgent from '../agents/triage';
export async function run({ init, payload }) {
  const s = await (await init(triageAgent)).session();
  return (await s.prompt(payload, { result })).data;
}
Why two abstractions the decision trail · changelog & docs
1 · Two shapes of work
Open-ended tasks need the model to choose each step; bounded tasks need your exact steps. Flue runs both on one core instead of making you pick.
1.0-beta rationale
2 · Split by lifecycle
Capabilities match; lifecycles differ. An agent keeps a persistent session; a workflow records one finite run — a runId. So “runs are workflow-only.”
AGENTS.md · changelog 0.5
3 · Tightened by use
Shipped first as Astro's repo workflows, then aligned: parallel define* names, named sessions removed, and now every workflow wraps an agent — defineWorkflow({ agent, action }).
