Inside Flue.

The harness as a first-class build target. You write the agent once; you bring the runtime. A config object becomes an HTTP handler that deploys to a dozen targets — state is an event log, not a platform.
An agent is a function createAgent()
// agent/triage.ts import { createAgent, local } from '@flue/runtime'; import { replyToIssue } from './tools'; import triage from './triage.md' with { type: 'skill' }; import instructions from './agent.md' with { type: 'text' }; export default createAgent(() => ({ model: 'anthropic/claude-sonnet-4-6', // any provider tools: [replyToIssue], skills: [triage], // Markdown, on demand sandbox: local(), // or just-bash / remote channels: [slack(), github()], instructions, }));
Served as an HTTP route by AgentRouteHandler — POST /agents/triage/:id. Two modes: Agents (open-ended) and Workflows — run(), where your code guides the reasoning.

What's distinctive vs the field
Runtime-agnostic
Agents are HTTP handlers — deploy to a dozen targets.
Event-sourced
Durable Streams: accepted work is never lost; sessions resume; clients reconnect.
BYO store
connect() → executionStore · runStore · eventStreamStore.
Model-agnostic
a provider-prefixed string; swap the core, keep the harness.

The takeaway: Flue is the framework; the runtime is your choice. The inverse of a platform-native harness.
